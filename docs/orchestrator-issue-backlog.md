# Orchestrator Issue Backlog

这份文档记录当前互动小说生成链路里已经暴露、并且需要继续优化的问题方向。它不是最终方案，也不是修复任务拆解；它的用途是把问题说清楚，避免后续讨论在 `Director`、`Narrator`、`storyline`、`requiredContent`、`constraints` 等术语之间漂移。

本文以最近几组 `membackend` 和 `hybrid-dual-layer-memory` run 的人工复核为主。旧的优化日志只能作为参考，不能反过来决定问题分类。后续如果要把旧日志里的观点合并进来，需要逐点确认它是否能解释当前 case。

## 判断原则

分析问题时先看玩家实际看到的内容，再回到日志链路追原因。不能先拿隐藏剧本、角色卡或内部状态去判定玩家可见文本有错。

一次问题分析至少要区分三层：

| 层级 | 要回答的问题 |
| --- | --- |
| 玩家可见问题 | 玩家实际看到了什么不一致、重复、拖沓或割裂的内容。 |
| 直接原因 | 哪个环节把问题写进了最终结果，例如 Director 的安排、Narrator 的正文、Choice 的选项或状态写回。 |
| 根本原因 | 为什么系统允许这个问题发生，例如信息边界不清、约束重复、输入处理策略缺失、节奏控制不足或承接防线薄弱。 |

不要把 “Director 写错了”“Narrator 能力不行”“Choice 生成不好” 当成最终结论。这些通常只是直接原因或现象描述。

## 字段和职责边界

当前讨论里最容易混淆的是 `requiredContent` 和 `constraints`。它们不是同一个东西。

| 字段 | 来源和位置 | 语义 |
| --- | --- | --- |
| `currentStoryline.constraints` | `storyline` 里的长期字段。 | 当前故事线需要持续遵守的约束，会被传给 Director 和 Choice。 |
| `requiredContent` | Director 本轮 `plotPoint` 输出。 | 本轮必须写进正文的固定演出、固定对白、资源调用或关键顺序，不是 storyline 字段。 |
| `currentTurnConstraints` | Director 本轮 `plotPoint` 输出。 | 只约束当前 turn 的呈现。 |
| `currentStorylineConstraints` | Director 本轮 `plotPoint` 输出。 | 需要持续保留到当前故事线后续的约束，写回后会并入下一轮的 `currentStoryline.constraints`。 |

因此，后续说 “约束重复” 时要说清楚是哪一层重复。当前更准确的问题是：Director 新生成的 `currentStorylineConstraints` 不应该和已有 `currentStoryline.constraints` 语义重复。

## 当前需要解决的问题

### 1. 玩家输入处理策略不足

Director 需要把玩家输入当作本轮剧情的起因，而不是必须照做的指令。它应该先判断玩家是在行动、探索、表达、对抗还是脱轨，然后在世界规则、角色动机和当前故事线约束下安排后果。

当前问题是，部分 prompt 只保留了 `playerIntent` 这个输出字段，但没有充分说明分类之后应该如何处理玩家输入。这样容易让 Director 只是表面回应玩家输入，而不是把玩家输入转化成合理的剧情推进。

这个问题和剧情节奏有关，但不是同一个问题。玩家输入处理回答的是“本轮为什么这样推进，以及怎样回应玩家”；剧情节奏回答的是“本轮推进多少，是否拖沓或过碎”。

### 2. Storyline 与 Director 输出的职责边界不够稳定

`storyline` 提供当前故事线目标、已完成进展、可参考素材和长期约束；Director 每轮输出本轮安排、固定内容、一次性约束和需要持久化的新约束。两者边界如果不清，就会出现几类问题：

- `requiredContent` 被用得太碎，固定剧情被拆成多轮小片段。
- `currentStorylineConstraints` 写回后变成长期约束，但没有和已有约束区分清楚。
- `storyline.content` 里“可参考素材”和“已完成进展”的边界不够醒目，后续 worker 可能重复已经演过的内容。
- `storyline.summary` 虽然记录了已发生内容，但没有稳定地变成“下一轮从这里之后继续”的硬承接锚点。

这里的核心不是简单增加更多字段，而是让每层信息的用途更明确：什么是本轮必须消费的内容，什么是长期禁止越界的信息，什么只是可参考素材，什么已经被玩家看到、不能重复演。

### 3. Storyline 约束重复和过期约束压力

当前已经观察到语义接近的约束可能重复出现，例如同一类 PTSD 表现约束或“不要提前揭示暗街深处信息”的近似表达。精确文本去重无法处理这类语义重复。

重复约束本身不一定立刻造成玩家可见错误，但它会让 prompt 里某些限制被反复强化，压过当前情景的真实变化。更麻烦的是，剧情已经切换或片段已经结束后，旧约束如果继续留在当前故事线里，可能限制后续情节展开。

这个问题目前先描述为“约束重复和生命周期不清”，不要直接跳到某个具体修复方案。后续需要通过 case 判断：哪些约束应该长期保留，哪些应该只保留一轮，哪些应该随着故事线推进自动失效。

### 4. Narrator 正文承接和已说内容消费不足

Narrator 有时没有从上一轮玩家可见正文的末尾继续，而是回退到前文已经写过的动作、场景或表述，再演一遍相似内容。这个问题包括两种表现：

- 续写锚点回退：正文不是接着上一轮末尾写，而是回到前面某个场景状态重新铺。
- 跨轮表述重复：相邻或相隔几轮出现几乎相同的行动、条件交换、问答或描述。

例如此前复核过的裁缝相关 case 中，角色在不同轮次反复表达“把东西带回来，我就告诉你纸条上的人在哪”这一类交易条件。另一些 case 中，Narrator 会重复进入、敲门、反问或重新确认已经回答过的信息。

这个问题不能简单归为 “Narrator 能力弱”。如果 Director 的安排本身大体合理，Narrator 仍然应该作为正文最后防线，严格遵守最近可见事实和上一轮收尾位置。但如果 Director 没有明确哪些内容已经被消费，或者 prompt 把旧素材和已完成进展混在一起，Narrator 的重复倾向就会被放大。

### 5. 故事线推进节奏过慢、过碎

当前比较核心的问题之一是剧情推进节奏。部分 run 里 Director 每轮给出的情节安排过少，导致固定剧情被拆成多个很小的片段，自由发挥部分也只推进一个局部动作或一句对话。最终阅读体验会变得碎片化、拖沓，像在同一个剧情片段里原地打转。

固定剧情消费应该归到这个问题下，而不是单独列成一个顶层问题。我们的当前判断是：固定剧情一旦触发，原则上应该在一个 turn 里消费完。否则系统很容易在后续 turn 里继续围绕同一个固定演出拆分、回放或补尾巴。

Choice 也参与这个问题。如果一个 storyline 超过若干轮没有明显变化，选项不应该继续只提供原地追问或细节观察，而应该逐步引导玩家结束当前片段、进入下一个剧情动作。Director 层面则需要在回应玩家输入的同时推进当前 storyline，而不是只处理眼前一句话。

### 6. 情节切换过度依赖状态机，叙事闭环不足

状态机可以决定当前进入哪个节点，但它不能替代玩家可见叙事里的过渡和收束。当前风险有两个方向：

- 状态机已经触发新情节，但原情节在正文里没有自然结束，玩家会感觉情节突然跳走。
- 原情节已经在玩家可见层面结束，但旧 storyline 或旧约束仍然继续影响后续生成，导致剧情不自然地回头。

这个问题和推进节奏相连。真正需要解决的是“状态切换”和“玩家可见情节切换”之间的对齐，而不是只在状态机里增加更多分支。

## 暂不直接采纳的表述

旧优化日志里有一些有参考价值的说法，比如 fixed beat 的 pending、active、consumed、invalidated 状态，以及 storyline lifecycle 的显式管理。但在当前问题清单里，暂时不要把它们作为问题定义本身。

更准确的当前表述是：

- 固定剧情拆太碎、重复消费，先归入故事线推进节奏问题。
- 固定剧情是否需要显式 consumed 状态，是后续可能的实现机制，不是今天已经确定的问题分类。
- 约束是否需要生命周期管理，也需要结合具体 case 再决定，不能从字段设计直接倒推出修复方案。

## 后续复核时优先看的证据

针对上面这些问题，复核 case 时优先看这些 artifact：

| 问题方向 | 优先证据 |
| --- | --- |
| 玩家输入处理策略 | Director system prompt、`06a-director-prompt.md`、Director output 里的 `playerIntent`、本轮玩家输入和最终正文。 |
| 字段职责边界 | `03-story-state.json` 的 `currentStoryline`、Director output 的 `requiredContent`、`currentTurnConstraints` 和 `currentStorylineConstraints`。 |
| 约束重复 | 连续多轮 `03-story-state.json`、Director output、写回后的 `05-runtime-after.json`。 |
| 正文承接重复 | 前一轮和当前轮 `04-output.json`、`06b-narrator-prompt.md`、Director output 的 `beats` 和 `requiredContent`。 |
| 推进节奏 | 连续多轮玩家可见正文、Director output 的 `summary`、`beats`、`requiredContent`、`pacing`，以及 Choice 输出。 |
| 情节切换 | `02-script-state.json`、`03-story-state.json`、`05-runtime-after.json`、当前和相邻几轮玩家可见正文。 |

每个问题都应该先用玩家可见内容确认是否成立，再使用内部 artifact 解释为什么发生。
