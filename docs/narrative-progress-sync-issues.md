# Narrative Progress Sync Issues

这份文档总结最近多组 run 中反复出现的叙事进度同步问题。它关注的是玩家实际体验到的现象，而不是具体修复方案。后续讨论实现时，可以把这里的问题分类当作共同语境，避免把“事实不一致”“重复演出”“状态机跳转”“剧情节奏”混成一团。

核心判断是：当前最影响体验的问题，不只是模型偶尔写错事实，而是玩家可见正文、选项、Director 节点和 runtime 状态之间没有稳定对齐。很多看似不同的问题，都是这几层不同步之后在不同位置冒出来的症状。

需要特别说明的是，这份文档不是只讨论“剧情切换”。剧情切换是其中一类高影响问题，但有些现象发生在同一个剧情节点内部，例如同一场戏重复演出、正文已经完成某个动作但选项仍然停在动作之前、固定演出没有完整落地。这些问题和跨节点切换有关联，但不能都归因于剧情切换。

## 问题总览

可以把当前现象概括为一句话：

> 玩家看到的剧情进度，和系统内部认定的剧情进度，经常不是同一个进度。

这种错位会产生几类直接体验：

- 剧情突然切到另一条线，缺少必要过渡。
- 正文已经演过的动作，下一轮又被当成未发生。
- 状态已经进入后续节点，但正文还没有铺好前置情节。
- 固定演出被拆得过碎，关键桥段没有完整落地。
- 软节点停留过久，同一组信息被反复追问或复述。
- 选项把未建立的前提暴露给玩家，或者让玩家重复已经完成的动作。
- 未选择路径或隐藏信息提前进入正文，后续又被当成已发生事实。

这些问题会让玩家感觉剧情一会儿快进，一会儿倒带，一会儿卡住，一会儿又默认自己经历了没有经历过的事。相比单个事实错误，这种进度错位更容易破坏互动小说的因果感。

## 和剧情切换的关系

下面这些问题和剧情切换直接相关：

- 剧情突然切线。
- 状态先行，正文未铺垫。
- 后续节点信息提前泄露，导致真正切换时缺少可见支撑。
- 已经演过跳转条件，但状态机没有触发，导致错过跳转。

下面这些问题不一定是剧情切换造成的，它们也可能发生在同一个剧情节点内部：

- 正文先行，状态未消费。
- 固定演出没有完整落地。
- 软节点滞留和重复演出。
- 选项重复已经完成的动作。

所以更准确的总问题不是“剧情切换错误”，而是“可见叙事进度和内部进度不同步”。剧情切换只是这个同步问题在节点边界上的表现。

## 现象一：剧情突然切线

玩家可见正文还在推进一条线，系统内部已经切到了另一条线。正文层面会表现为目标、地点、人物关系或任务重心突然改变。

典型体验是：玩家刚刚确认要继续寻找某个线索，下一轮却突然进入宴会、会面或另一段剧情。这个问题不一定是新节点本身错误，而是切换缺少可见过渡。玩家没有看到当前情节如何收束，也没有看到下一个情节为什么自然发生。

排查时需要同时看三层：

| 层级 | 需要确认的问题 |
| --- | --- |
| 玩家可见正文 | 上一轮结尾和当前轮开头是否自然承接。 |
| Director 输入输出 | 当前轮是否已经拿到新节点，并按新节点安排剧情。 |
| 状态写回 | beat 或 storyline 是否提前推进到玩家尚未到达的位置。 |

这个问题的关键不在于“状态机不能切”，而在于状态切换不能替代玩家可见叙事里的收束和过渡。

## 现象二：正文先行，状态未消费

这是最近比较值得单独命名的一类问题。正文已经演出了某个动作或节点结果，但 runtime 没有把它登记成已发生，后续选项或节点仍然把它当作未发生。

典型表现包括：

- 正文已经写玩家坐下，但选项仍然是“接受邀请”和“拒绝邀请”。
- 正文已经进入宴会厅，下一轮又从“你踏入宴会厅的那一刻”重新开始。
- 角色已经完成告别、退入阴影并关门，后续几轮又反复站在同一门口说同一组话。

这类问题不是普通复述。它说明玩家可见世界和系统状态已经分裂：

| 玩家视角 | 系统视角 |
| --- | --- |
| 我已经坐下。 | 坐下邀请还没有被选择或消费。 |
| 我已经进入宴会厅。 | 入场节点仍然可以重新演出。 |
| 这段告别已经结束。 | 当前软节点还没有明确结束。 |

这个现象会直接导致重复演出，也会导致后续跳转错过。因为后续真正到达对应节点时，系统不知道玩家已经看过这段内容，只能再次演出，或者继续围绕旧状态生成选项。

## 现象三：状态先行，正文未铺垫

这是“正文先行”的反面。系统内部已经推进到后续节点，但玩家可见正文还没有完成必要的前置演出。

玩家会看到一些缺少支撑的默认事实。例如某个角色、地点、任务或关系突然成为当前语境的一部分，但此前没有足够的可见铺垫。严重时，正文会把玩家没有选择、没有经历的路径写成已经发生。

这类问题经常被评测标成“无支撑跳跃”或“时空断裂”，但更底层的现象是：runtime 认为剧情已经到了某处，玩家可见叙事还没有走到那里。

## 现象四：固定演出没有完整落地

有些故事节点不是简单的剧情梗概，而是需要一段具体的戏。它包含必要的动作顺序、对白、观察对象、情绪变化和信息释放节奏。当前部分固定演出没有被完整转换成玩家可见正文。

常见表现有三种：

- 关键桥段被省略，只剩一两句概括。
- 必须发生的对白或动作被拆散到多个 turn，导致中间被玩家选择或其他状态打断。
- Director 给出的 beats 更像剧情摘要，Narrator 自行补写时漏掉了真正支撑跳转的动作。

这个问题和“模型有没有遵循 prompt”有关，但不能只归咎于 Narrator。当前 beats 里很多内容偏剧情线概括，不够像“讲戏”。当 Director 没有把这场戏具体讲清楚时，Narrator 就容易只抓到几个情节点，而不是完成一个可承接的场面。

## 现象五：软节点滞留和重复演出

相对自由的剧情节点里，系统有时没有明确判断“这段已经够了，可以往下走”。于是同一场景、同一组信息、同一个观察动作会持续出现。

玩家可见表现包括：

- 连续几轮围绕同一个物件观察，没有新的剧情状态变化。
- 角色重复回答已经回答过的问题。
- 告别、休息、等待、沉默这类收束动作反复出现，却迟迟不真正进入下一段。
- 选项继续提供原地追问或重复观察，而不是引导当前片段结束。

这个问题不等同于“自由节点不应该自由”。自由节点指的是编剧内容可以比较开放，但 Director 仍然需要把当前轮讲成一个实的叙述动作。否则自由会退化成原地打转。

## 现象六：选项层放大错位

Choice 不是单纯的用户界面，它会把当前系统认定的可行动作暴露给玩家。当前很多进度错位都会通过选项被放大。

常见问题有：

- 选项要求玩家执行正文已经完成的动作。
- 选项暴露正文尚未建立的角色、地点或任务。
- 选项把 Director 或内部剧本计划中的未来动作提前放到玩家面前。
- 选项重复上一轮已经问完的问题，下一轮 Narrator 又被迫重复回答。

一旦玩家选择了这样的选项，下一轮生成就会把这个错误前提固化成剧情。因此，重复演出有时不是 Narrator 单独造成的，而是 Choice 把旧状态或未来状态重新递给了 Narrator。

## 现象七：信息提前泄露或未选路径被写成已发生

有些正文会提前使用后续节点、隐藏设定或玩家未选择路径中的信息。它不一定立刻表现为事实冲突，但会破坏剧情节点本应承担的信息释放和跳转作用。

典型表现包括：

- 玩家尚未见过某角色，正文却说玩家已经见过。
- 玩家选择离开某场合，后续却默认玩家在场合内完成了关键会面。
- 后续节点的信息提前混入当前对话，导致真正到节点时缺少新信息可演。
- 隐藏设定被当成玩家可见事实使用，削弱悬疑感。

这类问题会和“正文先行，状态未消费”互相叠加。正文提前演了未来节点的信息，但状态机没有通过正式路径跳转；后面到了节点时，系统既不知道前面已经泄露，又仍然需要按原节点安排继续推进。

## 常见评测标签和真实现象的关系

一致性评测里的标签是定位入口，不一定等于根因分类。同一种底层错位，可能被标成不同问题。

| 评测标签 | 常见玩家可见现象 | 可能对应的底层错位 |
| --- | --- | --- |
| `repeated-scene` | 同一动作、对白或场面重复演出。 | 正文先行但状态未消费；软节点没有结束判断；选项重复旧动作。 |
| `unsupported-jump` | 没有铺垫地默认某事已经发生。 | 状态先行但正文未铺垫；未选择路径被写成已发生；隐藏信息提前泄露。 |
| `space-time-break` | 人物位置、动作顺序或场景状态回退。 | Narrator 承接失败；旧节点内容被重新演出；状态和可见正文不一致。 |
| `user-input-ignored` | 玩家选择没有被剧情承接。 | 当前节点安排压过玩家输入；状态机切线先于正文承接。 |
| `quality-regression` | 轻微重复、表述矛盾或节奏拖沓。 | 可能是局部文本问题，也可能是进度错位的低强度表现。 |

后续人工分析时，不要只按标签归类。应该先确认玩家实际看到的问题，再判断它属于哪种进度错位。

## 分析时优先看的证据

排查这类问题时，建议先从玩家可见时间线开始，再回到 turn 日志找系统进度。

| 需要判断的问题 | 优先查看的文件 |
| --- | --- |
| 玩家实际看到了什么 | `consistency-review/<run-id>/visible-timeline.jsonl`、对应 `turn-XX/04-output.json`。 |
| 当前轮系统认为处在哪个节点 | `turn-XX/01-summary.json`、`turn-XX/02-script-state.json`。 |
| 当前故事线和候选动作是什么 | `turn-XX/03-story-state.json`。 |
| Director 是否已经切到新剧情 | `turn-XX/06a-director-prompt.md`、`turn-XX/06-llm-calls.json`。 |
| Narrator 是否拿到了正确承接信息 | `turn-XX/06b-narrator-prompt.md`。 |
| Choice 是否重复或泄露动作 | `turn-XX/06c-choice-prompt.md`、`turn-XX/04-output.json`。 |
| 本轮后状态是否消费了动作 | `turn-XX/05-runtime-after.json`。 |

最小判断链路可以是：

1. 用玩家可见正文确认问题是否真实成立。
2. 看上一轮结尾和当前轮开头，确认是跳跃、回退、重复还是泄露。
3. 看 summary 里的 beat、storyline 和 choices，判断系统内部进度是否和正文一致。
4. 如果内部状态和正文不同步，再追 Director、Narrator、Choice 哪一层最早把错位写出来。

## 当前结论

目前最值得关注的不是某一个 worker 单点写错，而是四层进度同步问题：

| 层级 | 应该承担的职责 | 当前风险 |
| --- | --- | --- |
| Director | 决定当前轮该推进哪段戏，以及如何承接玩家输入。 | 可能过早切线、讲戏不足，或只给剧情梗概。 |
| Narrator | 把当前安排写成严格承接上一轮的玩家可见正文。 | 可能回退、重复、提前演未来节点，或漏掉关键桥段。 |
| Choice | 给出玩家当前可见状态下合理的下一步行动。 | 可能重复已完成动作，或暴露未建立前提。 |
| Runtime state | 记录哪些动作、节点和剧情进度已经发生。 | 可能没有消费正文已经演出的动作，或比正文更早推进。 |

因此，后续讨论修复时需要避免只说“Director 有问题”或“Narrator 能力不够”。更准确的问题是：系统缺少一个稳定机制，让玩家可见剧情进度和内部剧情进度在每一轮之后重新对齐。
